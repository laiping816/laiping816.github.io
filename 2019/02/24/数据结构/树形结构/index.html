树形结构

  • 树
    • 二分搜索树

    • 堆；并查集
    • 线段树；Trie（字典树，前缀树）

二分搜索树

动态数据结构

  • 特点

    • 每个节点的值 > 其左子树所有节点值，< 右子树
    • 每棵子树也是二分搜索树
    • 自定义类，必须提供比较规则

  • 遍历：O(n)

    • 深度优先遍历（一扎到底）

      • 前序（node, node.left, node.right）
        • 一般只是想遍历执行操作（或输出结果）可选用先序遍历
      • 中序（node.left, node, node.right）
        • 符合从小到大（或从大到小）顺序的，故要遍历输出排序好的结果需要使用中序遍历
      • 后续（node.left, node.right, node）
        • 肯定已经遍历过该节点的左右子节点，故适用于要进行破坏性操作的情况，比如删除所有节点

    • 广度优先遍历（更快找到问题的解 - 最短路径）

      • 层序

集合(Set)

元素不重复

  • 实现方式

    • 二分搜索树（有序）

      h（层数1.2.3…）对应有2^(h - 1)个结点。满二叉树结点数n = 2^h - 1

      h = log(n + 1)

      • 增删查的复杂度：O(h) = O(logn)

    • 链表（无序）

      • 复杂度O(n)效率低
      • 基于哈希表效率高

当h = n时，二分搜索树即退化成链表。

解决方法：平衡二叉树

映射(Map)

  • 时间复杂度
    • 链表
      • 增删改查O(n)
    • 二分搜索树
      • 增删改查O(h) = O(logn)

集合，完全可以通过映射实现，只要把value置为null即可

堆

优先队列

  • 实现

    • 普通线性结构
      • 入队：O（1）
      • 出队：O（n）

    • 顺序线性结构

      • 入队：O（n）
      • 出队：O（1）

    • 堆

      • 入队：O（logn）
      • 出队：O（logn）

二叉堆

  • 特点

    • 完全二叉树（区别满二叉树）：最后层上面一层一定是满的，最后1层叶子结点一定在最左边
    • 堆中某个节点的值总是 <= 其父节点的值 （最大堆）

  • 存储结构

    • 树(略)

    • 数组

      注意：为了后面求parent和child数组0号位置空出

    保留第一个位置：

  • 操作堆

    • 添加(Sift Up堆特有称谓)

      • 先加在数组最后面
      • 然后跟父亲比较，一层层比上去直到根节点或者小于等于其父节点

    • 取出堆中最大元素(Sift Down)

    • replace(取出最大并替换)

    • heapify(将任意数组整理成堆的形状)

      • O(n)

      • 关键：找到最后1个非叶子结点，一直上去到根一直做下沉操作

        • 定位最后1个非叶子结点，就是最后1个节点的父节点

索引堆

线段树

  • 引入

    • 区间染色
    • 区间查询

    对于给定区间

    更新：更新区间中的一个元素或者一个区间的值

    查询：一个区间[i, j]的最大值，最小值，或者区间数字和

  • 特点

    • 平衡二叉树（最大和最小深度差距1）

并查集

Union Find

  • 复杂度：O(log*n) 近乎是O(1)

  • 场景

    • 处理连接问题Connectivity Problem
    • 路径问题
    • 集合并集

  • 特点

    • 孩子指向父亲

实现

  • 用数组，查O(1)，但是合并O(n)

  • 树形，查O(logn)，合并O(logn)

路径优化

  • 基于size优化

  • 基于rank优化(更合理)

    基于size，在如下情况就不适用

  • 路径压缩(多调用几次也能达到最优效果)

    发生在find方法过程

    基于性能考虑不维护rank，但rank仍可作为参考

  • 最大化路径压缩(递归耗费性能，并非最优)

平衡二叉树

解决普通二分搜索树，最差插入退化成链表

  • 定义：任意节点，左右子树的高度差不超过1

AVL

  • 计算节点高度和平衡因子(即左右子树高度差)

  • 左、右旋转

    • LL不平衡

      • 💡解决：右旋转。证明略

    • RR：左旋转

    • LR：先对x左旋转转换为LL问题，在对y右旋转

    • RL：先对x右旋转转换为RR问题，在对y左旋转

  • 再添加和删除操作中检查平衡

红黑树

要点：

  1. 红黑树和2-3树等价关系
  2. 红黑树5条性质
  3. 牺牲平衡性，保持了”黑平衡”，统计性能更优
  • 算法导论中的定义 (最大高度：2logn 即每个节点都是3-节点)
    • 每个节点红的或黑的
    • 根节点是黑的
    • 每个叶子节点(最后的空节点)是黑的。定义了空的节点，为黑色
    • 如果一个节点是红的，那么它的孩子节点是黑的
    • 从任意一个节点到叶子节点，经过的黑色节点是一样的(保持了”黑平衡”的二叉树)

2-3树

  • 特点

    • 满足二分搜索树的基本性质
    • 节点可以存放一个元素（2个孩子）或者两个元素(3个孩子)
    • 绝对平衡，任意节点左右子树高度相同

  • 2-3树维护绝对平衡

    添加节点不会添加到空的位置

    • 插入2-节点，就普通融合构成3-节点即可

    • 插入3-节点

      • 父亲节点是2-节点

      • 父亲节点是3-节点

        最后一步退化继续向上融合后，在向上递归直到遇到2-节点或者根节点停止

转换与等价性

  • 普通2-节点就是黑色
  • 3-节点中小的那个数为红色，大的仍为黑。因此所有红色节点都是左倾斜的

​ 例子：普通转换示例

添加元素

  • 插入到2-节点

    • 插入元素比2-节点小，就直接添加到左子树即可

    • 插入元素比2-节点大，需要左旋转

  • 插入到3-节点

    • 当前 > 3-节点，颜色翻转

    • 当前 < 3-节点，右旋转再颜色翻转

    • 当前在 3-节点2个数值中间，先左旋转再右旋转最后，颜色翻转

  • 最终add添加维护红黑树的过程

性能总结

  • 完全随机数据，普通BST即可。缺点：极端退化成链表(或高度不平衡)
  • 查询较多，AVL，但保持平衡开销比红黑树要高
  • 红黑树牺牲了平衡性(2logn的高度)，统计性能更优(综合增删改查所有的操作)

哈希表(散列表)

  • 经典设计思想：空间换时间

  • 什么是哈希表？

    • 例子

      ​ 例如单词（只含小写字母）字母频率统计，int[26] freq 就是一个哈希表。每一个字符都和一个索引相对应

    • 哈希函数 (“键”转换为”索引”)

      • ch - 'a'

        字符转换为索引的函数

    • 哈希冲突

哈希函数设计

  • 原则

    “键”转换为”索引”

    • 一致性：if a = b 则hash(a) = hash(b)也一定相等
    • 高效性：计算高效简便
    • 均匀性：哈希值均匀分布

  • 对象

    • 大整数

      • %M，模一个素数(数论相关)，素数选择上网查，该素数即哈希表大小

    • 浮点数

      • 内存空间当做int、long来处理转换为大整数的问题

    • 字符串

      • 看做是1个B(base用户可选，如全是小写字母，就为26)进制的大整形

    • 符合类型

      • 可以套用字符串的处理方法，每个字符可以对应一个字段，Java中进制B取了31。

  • Java中的hashCode

    注意：hashCode只是把”键”转化成了一个整型值，具体要转换为”索引”，还需要%M，这个需要在自己设计的哈希表的类中内部逻辑来完成

哈希冲突处理

  • 链地址法(Seperate Chaining)

    • 保证hashCode为正数 &0x7fffffff，取模

    • 每一个索引存储的查找表，用树替代

    注意：转成红黑树也是有条件的，如果Key不具有可比较性那不能转

  • 开放地址法

    • 线性探测：效率低
    • 平方探测：1+4+9+16…
    • 二次哈希
  • 再哈希法
  • Coalesced Hashing：综合链地址法和开放地址发

复杂度分析

  • 平均复杂度

    总共M个地址，如果放入哈希表的元素为N，哈希碰撞导致平均复杂度如下

    • 链表：O(N/M)
    • 平衡树：O(log(N/M))

  • 优化趋于均摊复杂 O(1)

    • 动态空间处理

      • 平均每个地址承载的元素多过一定程度，即扩容 N / M >= upperTol
      • 平均每个个地址…少…，缩容N / M < lowerTol

    • 容量变换后M可能不是素数

      • 构建素数数组，通过索引增加来改变M大小

B-树

  • 为什么不用二分搜索或者红黑树？

    ​ 磁盘IO，db索引存储在disk上，数据量大，索引可能有几个G。当查询时，显然不可能把整个索引全部加载到内存。能做只有逐一加载每个磁盘页(索引树的节点)。

    注意：实际是以磁盘块为单位读取，而InnoDB存储引擎是以页为最小单位管理，方便起见称磁盘页

    • eg：如果拿h=4的二分搜索找10

      • 第一次磁盘IO，找9
      • 第2次，找13
      • 第3次，找11
      • 第4次，找到10

      磁盘IO的次数是由树的高度所决定

      因此为了减少磁盘IO的次数，核心思想：就需要把原来”瘦高“的树结构，变得”矮胖“

  • 特点

    多路平衡二叉树，每个节点最多包含k个孩子(称”B树的阶”)。k的大小取决于磁盘页的大小

    m阶B树的特征：

    • 根节点至少2个子女
    • 每个中间节点都包含k-1个元素和k个孩子，(其中$\frac{m}{2} \leq k \leq m$)
    • 和上面一条一样，每个叶子节点都包含k-1个元素，(其中$\frac{m}{2} \leq k \leq m$)
    • 所有叶子叶子节点都位于同一层

    • 每个节点中的元素从小到大排列，节点当中k-1个元素正好是k个孩子包含的元素的值域分划。

    • eg：

      • 查找操作：3阶B-树为例（就是2-3树），找5的过程

      整个流程，查询次数一点也不必二分搜索树少，但相比磁盘IO的速度，内存中的比较耗时几乎可以忽略。因此只要树的高度够低，IO次数足够少，就能提高性能。

      • 添加、删除操作

        和2-3树添加数据一样，保持完全平衡

  • 应用

    • 文件系统以及部分数据库索引，比如著名的MongoDB，大部分关系型数据库，则使用B+树作为索引

B+树

  • 特点

    • 有k个子树的中间节点包含有k个元素（B树中是k-1个元素），每个元素不保存数据，只用来索引，所有数据都保存在叶子节点。
    • 所有的叶子结点中包含了全部元素的信息，及指向含这些元素记录的指针，且叶子结点本身依关键字的大小自小而大顺序链接。
    • 所有的中间节点元素都同时存在于子节点，在子节点元素中是最大（或最小）元素。

    注意：根节点的最大元素，也是是整个B+树的最大元素。优化无论插入或者删除元素，记得维护。

  • 卫星数据

    实际就是数据记录行，在B+树中只有叶子节点带卫星数据，其他中间节点只是索引

    • 聚集索引：叶子节点直接包含卫星数据
    • 非聚集：包含的是卫星数据的指针

  • 优点

    • 单行查询：eg：上面查3

      (8,15) -> (2, 5, 8) -> (3, 5)，看似过程和B-tree差不多

      • B+ tree中间节点无卫星数据，同样大小磁盘页可容纳更多节点元素(更”矮胖”)、IO少
      • 查询性能稳定，一定会找到叶子节点，而B-tree则不稳定

    • 范围查询简便

      B-tree中要依靠中序遍历，复杂。

      B+tree只要找到下限，根据链表指针找到上限即可
